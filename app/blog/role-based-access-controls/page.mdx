## Who Can Access What?

As your application grows, managing user permissions becomes increasingly complex. Hardcoding `if (user.role === 'admin')` statements all over your codebase is a recipe for disaster. It's brittle, hard to manage, and makes it impossible to get a clear picture of who has access to what. A simple change, like introducing a new "editor" role, can require a massive refactoring effort.

Our **Role-Based Access Control (RBAC) Manager** provides a centralized and flexible way to manage permissions, decoupling your authorization logic from your application code.

## What is an RBAC Manager?

An RBAC Manager is a tool that allows you to implement a role-based access control system. The core idea is simple:
1.  **Permissions** are individual actions (e.g., `read:document`, `write:document`, `delete:user`).
2.  **Roles** are collections of permissions (e.g., an "editor" role might have `read:document` and `write:document` permissions).
3.  **Users** are assigned one or more roles.

Your application can then check if a user has a specific permission, without needing to know about their role. This makes your authorization logic clean, scalable, and easy to manage.

## A Common Use Case: A Collaborative Blogging Platform

You are building a platform where users can write and publish articles. You need different levels of access for different types of users.

1.  **Define Roles and Permissions:** In the RBAC Manager, you define the following:
    *   **Permissions:** `create:post`, `edit:own_post`, `edit:any_post`, `delete:post`, `publish:post`.
    *   **Roles:**
        *   **Contributor:** Can `create:post` and `edit:own_post`.
        *   **Editor:** Has all Contributor permissions, plus `edit:any_post` and `publish:post`.
        *   **Admin:** Has all Editor permissions, plus `delete:post`.

2.  **Assign Roles to Users:** When a new user signs up, they are assigned the "Contributor" role by default. You can manually promote a user to "Editor" or "Admin" through your application's admin panel, which calls the RBAC Manager's API.

3.  **Protect Your API Endpoints:** In your backend code, you protect your routes by checking for the required permissions.

    ```javascript
    // Example using Express.js middleware
    app.post('/posts', hasPermission('create:post'), (req, res) => {
      // ... logic to create a post
    });

    app.put('/posts/:id', hasPermission('edit:any_post'), (req, res) => {
      // ... logic to edit any post
    });

    app.delete('/posts/:id', hasPermission('delete:post'), (req, res) => {
      // ... logic to delete a post
    });
    ```

4.  **Result:** Your authorization logic is clean and centralized. If you later decide to add a "Moderator" role that can `delete:post` but not `publish:post`, you can create it in the RBAC Manager in seconds, without touching your application code.
